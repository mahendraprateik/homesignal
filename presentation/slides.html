The Problem

What exists today

  • Redfin Data Center — raw tables, no intelligence layer
  • Zillow / Datadoor — static dashboards, no conversational access
  • ChatGPT — no access to current housing data, hallucinates numbers

What I built

  • Live dashboard with 5 key metrics across 20 metros
  • AI-generated tooltips and daily briefs — not just data, insights
  • Conversational chat grounded in real, cited data
  • An agentic engine that chooses between vector search and SQL tools

HomeSignal = Redfin's data + FRED economic indicators + Claude's reasoning — automated daily, deployed on the web, with evaluations and feedback loops.

Vector Store Design

Three Document Types in ChromaDB

TypeCountExampleWhy It Exists
market_data 1 per (region, month) "Seattle, Mar 2026: Median $850K, 45 DOM, 8,200 inventory..." All 7 geo levels embedded — metro, city, county, state, ZIP, neighborhood, national
metro_trend 1 per region "Seattle: $750K to $850K (+13.3%), rising trend over 18 months" Pre-computed so "how has Seattle changed?" doesn't need 12 retrievals
metric_definition 1 All metric definitions from the semantic model Grounds the embedding space — anchors what "days on market" means

Second Collection: Web Context

CollectionSourcesPurpose
housing_market Redfin metrics (all geo levels) Market data snapshots, trend summaries, metric definitions
housing_context 7 web sources (Redfin News, Zillow Research, Freddie Mac, Fed, BLS, Census, Redfin Migration) Enriched article chunks — market narratives, forecasts, macro context

Context ingestion pipeline: discover URLs → extract articles → semantic chunking → enrich (signals, drivers, metrics) → summarize → deduplicate → embed. Content-hash dedup prevents re-inserting existing chunks.

Embedding Model

  • BAAI/bge-small-en-v1.5 — 384 dimensions
  • Runs locally — no API calls, no cost, no external dependency
  • Same model for both collections — consistent embedding space

Rebuild Strategy

  • housing_market: cleared and rebuilt every run — fast, prevents stale docs
  • housing_context: incremental — deduplicates against existing, only inserts new chunks

RAG Engine

Query Flow

User Question
  ─▶ Metro Detection (alias map, longest-first matching)
  ─▶ Guardrail Check (future predictions, property valuations)
  ─▶ ChromaDB Retrieval (top-5 per metro, deduplicated)
  ─▶ Context Assembly (retrieved docs + metric definitions)
  ─▶ Claude Opus (temp 0.2, 4096 tokens)
  ─▶ Citation Enforcement ([1], [2] markers)
  ─▶ Confidence Scoring (high / medium / low)

Metro Detection

  • Alias map: "Phoenix" / "Phoenix AZ" / full name all resolve to canonical
  • Sorted longest-first — prevents substring shadowing
  • Follow-up inference: scans history for most recent metro

Guardrails

  • Future predictions blocked — "forecast", "predict", "2027"
  • Property valuations blocked — "what is 123 Main St worth"
  • Keyword heuristic: fast, no training data needed
  • Current accuracy: 60% — known weakness, classifier would be better

Confidence Scoring

  • ≥3 market docs = high
  • ≥1 = medium
  • 0 = low (early return: "I don't have enough data")

Chat Engine — The Agent

The most architecturally interesting component. Combines two retrieval paradigms in one agentic loop.

How It Works

1. RAG retrieval first
   (context docs always injected)
        │
2. Claude sees context + 4 SQL tools
        │
3. Tool-use loop (max 3 iterations)
   Claude calls tool ─▶ gets result
   ─▶ reasons further ─▶ may call again
        │
4. Merge sources (RAG + SQL)
   ─▶ Return cited answer

Why Hybrid?

  • RAG alone can't answer "which metro has the highest inventory?" — needs SQL aggregation
  • SQL alone can't answer "what's driving Phoenix's market?" — needs narrative context
  • The LLM is the router — it decides which tools to call

The 4 SQL Tools

ToolPurposeSafety
query_latest_metricsCurrent snapshot for a metroMetro alias validation
query_mortgage_rate30yr fixed rateNo input needed
compare_metrosSide-by-side (max 5)Per-metro validation
top_metros_by_metricRankingsMetric allowlist + limit clamp

Agent Safety & Termination

  • Max 3 iterations — prevents infinite tool-use loops
  • Stops when stop_reason != "tool_use"
  • All SQL uses parameterized queries — no injection risk
  • Dynamic columns validated against semantic model allowlist

Token & Memory Management

Token Budget

ParameterValueRationale
Max output tokens4,096Enough for detailed answers with tables & citations
Temperature0.2Low hallucination risk, natural prose (0.0 feels robotic)
Retrieval top-K5 per metroBalances context relevance vs. token cost
Tooltip max tokens60One sentence — Haiku, fast & cheap

Model Selection by Task

TaskModelWhy
Chat & Daily BriefOpusComplex reasoning, tool use, synthesis
Chart TooltipsHaiku60 tokens — doesn't need Opus. 10x cheaper, faster
Eval JudgeHaikuScoring is structured — deterministic at temp 0.0

Conversation Memory

  • 6 turns (12 messages) retained per conversation
  • Older turns dropped to control input token budget
  • Follow-up inference: scans history reverse-chronologically to find most recent metro
  • Chat history capped at 50 messages per metro in session state

Thread Safety

  • Engine singletons shared across Streamlit sessions — avoids re-loading models per request
  • threading.local() for per-request tool sources — prevents cross-session data leakage
  • Double-checked locking on singleton initialization

Caching Layers

  • Metric definition doc cached at __init__ — no ChromaDB round-trip per query
  • Tooltips & briefs cached in SQLite — subsequent visits are instant
  • Streamlit @cache_data with TTLs (5 min to 1 hour)

Cloud Deployment

Daily Automated Refresh

Cloud Scheduler (daily 3 AM PST)
  ─▶ Pub/Sub
  ─▶ Cloud Build
  ─▶ cloud_refresh_job.py
      ├── Check freshness thresholds
      │   (FRED: 8 days, Redfin: 35 days)
      ├── Ingest if stale
      ├── Rebuild ChromaDB vectors
      └── Package snapshot (tar.gz)
          ─▶ Upload to GCS
              + SHA256 checksum
              + latest.json manifest

Cloud Run (Always-On)

On startup / every 15 min:
  ─▶ Check GCS latest.json
  ─▶ If new snapshot:
      ├── Download & extract
      ├── Reset engine singletons
      └── Serve with fresh data

Why Snapshot-Based?

  • Cloud Run has ephemeral filesystem — can't persist SQLite across restarts
  • Snapshot packages everything into a portable archive
  • No managed database needed — GCS storage costs pennies
  • Simple, reliable, and cheaper than Cloud SQL (~$30/month saved)

Freshness Heuristic

  • FRED: stale if MAX(period_date) > 8 days (publishes weekly)
  • Redfin: stale if MAX(period_date) > 35 days (publishes monthly)
  • Stateless — just checks the database, no external dependency

Easy questions (factual lookups) score 7.14. Hard questions (reasoning) score 9.0. The bottleneck is retrieval precision, not LLM capability. SQL tools improved easy scores from 4.1 to 7.1.

Known Limitations

Guardrails (60% pass rate)

  • Keyword heuristic catches "predict", "forecast", "future" — but misses year-based references like "in 2027"
  • Some borderline questions slip through (e.g., "will inventory rise?" gets answered instead of blocked)
  • Fix: lightweight classifier trained on eval questions

Chat: RAG vs SQL Tool Selection

  • Claude sometimes answers from RAG vector context instead of calling the SQL tool for factual lookups
  • This can return data from a neighboring month or slightly different number
  • Fix: metadata-filtered retrieval + stronger tool-use prompting

AI Layer is Metro-Scoped

  • Pipeline ingests 7 geography levels into SQLite, but SQL tools and vectors currently serve metro-level only
  • ZIP, city, neighborhood queries return no results
  • Fix: add sub-metro SQL tools (data is already in the DB)

Chat is Metro-Filtered by Dropdown

  • Chat is scoped to the selected metro in the sidebar
  • Asking about Phoenix while Nashville is selected won't return Phoenix data
  • By design for session scoping — but could confuse users

Eval Suite

  • 25 questions — low statistical power
  • LLM-as-judge not validated against human scores
